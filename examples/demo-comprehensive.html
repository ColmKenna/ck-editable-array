Core Events

✅ Feature 2: Schema-Driven Validation

Validation Accessibility

Try this: Click "Edit" on the empty row. Notice the Save button is disabled, error messages appear, and the row has a red border. Fill in the fields to see errors clear in real-time. Screen readers will announce errors via ARIA live regions (role="alert", aria-live="polite"). The component uses JSON Schema for validation with required and minLength constraints.

How to recreate this sample:
  • Define a JSON schema (element.schema = { ... }) to describe required fields and constraints; the component runs validation automatically on change.
  • Place an element with data-error-summary for form-level messages and data-field-error="<field>" spans next to inputs so validation text has a home.
  • Bind inputs with data-bind so failed fields are flagged and the Save button stays disabled until the schema passes.
  • Customize validation visuals inside a scoped <style slot="styles"> block using selectors like .row-edit[data-row-invalid], .error-summary, and .field-error to match your brand.
  • CSS used in this sample: buttons are themed per action; .row-edit[data-row-invalid] adds red border/background; .error-summary and .field-error define the alert banner and inline errors.

🔗 Feature 3: Nested Property Binding

Core

Try this: Edit a row to see nested property paths in action. The component supports dot notation like data-bind="person.name" and data-bind="address.city" for complex data structures. This allows you to work with deeply nested objects without flattening your data model.

How to recreate this sample:
  • Bind nested properties directly with dot notation (data-bind="person.name", data-bind="address.city") inside your display and edit templates.
  • Seed the component with nested objects via element.data = [{ person: {...}, address: {...} }] to keep the shape consistent.
  • If users can add rows, set element.newItemFactory to return the same nested skeleton so new rows render with the correct keys.
  • Scope layout and button styling for this nested form inside the <style slot="styles"> block; the sample targets buttons locally so other demos remain unaffected.
  • CSS used in this sample: only button styling (base plus [data-action] variants) lives in the scoped style slot, leaving layout to the shared page styles.

🔒 Feature 4: Exclusive Locking

Core

Try this: Click "Edit" on any row. Notice that other rows become locked (grayed out and disabled with data-locked attribute) and the Add button is disabled. Only one row can be edited at a time, preventing data conflicts. The component uses the inert attribute for accessibility compliance.

How to recreate this sample:
  • No extra configuration is needed—when a row enters edit mode, the component automatically disables the Add button and marks other rows with data-locked plus inert.
  • Listen for beforetogglemode and aftertogglemode to track when locks engage or release (e.g., to log activity or block navigation).
  • Style locked states in a scoped <style slot="styles"> block using selectors like .display-content[data-locked] to dim and disable pointer events.
  • Keep your templates lightweight: a single data-action="toggle" button is enough to demonstrate locking; the component handles the focus trapping.
  • CSS used in this sample: .display-content[data-locked] applies opacity/grayscale and disables pointer events; button styles (base and per action) are scoped to this instance.

🎨 Feature 5: Custom Styling with CSS Parts

Theming

Try this: This example uses custom CSS injected via the slot="styles" to create a unique look with gradient backgrounds. The component provides CSS parts, data attributes, and classes for complete styling control. All styles are scoped to the shadow DOM, preventing conflicts.

How to recreate this sample:
  • Add a scoped <style slot="styles"> block inside the component to keep gradient rules from leaking to other demos.
  • Target your own template classes such as .row-display, .row-edit, and .row-display button to control backgrounds, borders, and hover states.
  • Use the exposed parts for outer chrome tweaks when needed: ::part(root), ::part(rows), and ::part(add-button) (e.g., ck-editable-array#demo5::part(root) { border: none; }).
  • Keep base button styles minimal and override per state to layer gradients with readable inputs, as shown with the purple/pink themes in this demo.
  • CSS used in this sample: gradient backgrounds for .row-display (purple) and .row-edit (pink), translucent buttons on display rows, light inputs on gradients, and per-action button overrides inside the scoped style slot.

📊 Feature 6: Readonly Mode

Core

Try this: This array is in readonly mode via the readonly attribute. Notice that the Add button is disabled, and all inputs are read-only. This is useful for displaying data that shouldn't be edited, such as historical records or audit logs. Users can still view details but cannot make changes.

How to recreate this sample:
  • Add the readonly attribute (or set element.readonly = true) to disable editing and hide destructive actions.
  • Keep your usual display/edit templates; the component automatically sets inputs to readonly and disables Add/Save controls.
  • Style the read-only state with a scoped <style slot="styles">, targeting selectors like button[disabled] or [readonly] inputs to tone down the UI.
  • Flip back to editable mode programmatically by setting readonly = false if you want to reuse the same markup for admin users.
  • CSS used in this sample: scoped button styling plus per-action colors; inputs inherit read-only styling from the component (no custom form CSS beyond buttons).

🪟 Feature 7: Modal Edit Mode

Core Accessibility

Try this: Click "Edit" on any row to open a modal overlay for editing. This keeps the rest of the list locked and accessible with proper ARIA attributes. The modal respects the `modal-edit` attribute and demonstrates how Save/Cancel behave in overlay mode.

How to recreate this sample:
  • Enable modal editing by adding the modal-edit attribute (or setting element.modalEdit = true); the edit template is automatically rendered inside a modal surface.
  • Reuse the same display/edit templates you use for inline mode—no markup changes are needed beyond enabling modal behavior.
  • Customize the overlay and surface inside <style slot="styles"> with selectors like .modal-overlay and .modal-surface, or via parts (::part(modal), ::part(modal-surface)) for padding, blur, and borders.
  • Listen for beforetogglemode / aftertogglemode to react to modal open/close (e.g., pausing background interactions or logging edits).
  • CSS used in this sample: scoped button theme, delete/restore visibility toggles, deleted-row badge styling, modal overlay blur/tint, modal surface shadow/border, and grid-based edit form spacing.
