<ck-editable-array>

<ck-editable-array> renders and manages an editable array of rows (primitive values or objects), using two light-DOM templates: one for display mode and one for edit mode. It wires up add/edit/save/cancel/delete/ restore behaviour and emits a datachanged event whenever the underlying data array changes.

Common uses include email/phone lists, address lines, and any repeatable form section that needs add/remove/edit capabilities while still posting data back as normal HTML form fields.

Installation

From npm

npm install @colmkenna/ck-webcomponents/ck-editable-array

// register once in your app entry
import '@colmkenna/ck-webcomponents/ck-editable-array';

From CDN

<script src="https://unpkg.com/@colmkenna/ck-webcomponents/ck-editable-array@latest/dist/ck-editable-array/ck-editable-array.min.js"></script>

In this repo, example pages use ../dist/ck-editable-array/ck-editable-array.esm.js.

Basic usage

Minimum setup: provide display and edit templates and assign an array to data. 

<ck-editable-array id="emails" name="emails">
  <template slot="display">
    <div>
      <span data-bind="value"></span>
      <button data-action="toggle">Edit</button>
      <button data-action="delete">Delete</button>
      <button data-action="restore">Restore</button>
    </div>
  </template>
  <template slot="edit">
    <div>
      <input data-bind="value" type="email" />
      <button data-action="save">Save</button>
      <button data-action="cancel">Cancel</button>
    </div>
  </template>
</ck-editable-array>

<script type="module">
  import '../dist/ck-editable-array/ck-editable-array.esm.js';
  const el = document.getElementById('emails');
  el.data = ['alice@example.com', 'bob@example.com'];
  el.newItemFactory = () => '';
  el.addEventListener('datachanged', e => console.log(e.detail.data));
</script>

Attributes & API

Attributes

NameTypeDefaultDescription
name string '' Base name for generated input names, e.g. person[0].name.
readonly boolean absent When present, disables all editing, adding, deleting, and restoring.
modal-edit boolean absent When present, renders the edit template in a modal dialog instead of inline. The modal uses ::part(modal) for the overlay and ::part(modal-surface) for the dialog surface, both fully styleable.

Properties

PropertyTypeDefaultDescription
data unknown[] [] Array of rows (primitives or objects). The getter returns a deep-cloned, public-safe snapshot.
schema object | null null JSON-schema-like object supporting required and properties[field].minLength.
newItemFactory () => EditableRow () => ({}) Factory for new rows when the Add button is clicked.
modalEdit boolean false Property mirror of the modal-edit attribute.

Events

EventWhenDetail
datachanged Whenever a row is added, saved, deleted, restored, or edited (outside of edit mode). { data: unknown[] }
beforetogglemode Right before a row switches between display and edit. { index, from, to }, cancelable via event.preventDefault().
aftertogglemode Immediately after a row has switched mode. { index, mode } 
const el = document.querySelector('ck-editable-array');
el.addEventListener('datachanged', (e) => {
  console.log('New data:', e.detail.data);
});

Templates & slots

Required templates

  • template[slot="display"] – row layout in read-only mode.
  • template[slot="edit"] – row layout in edit mode (usually with inputs).

Optional slots

  • template[slot="add-button"] – custom Add button area (use data-action="add").
  • style[slot="styles"] – CSS that is mirrored into the shadow DOM.
  • button[slot="button-edit"], button[slot="button-save"], etc. – custom action buttons.

Binding markers

  • data-bind="field" – bind text/value to a row field (supports nested paths).
  • data-action="toggle|save|cancel|delete|restore|add" – wire up built-in actions.
  • data-field-error="field" – container for validation messages for that field.
  • data-error-summary, data-error-count – optional row-level validation summary.

Custom action buttons

You can provide your own buttons inside templates and the light DOM; the element wires to buttons using data-action="...". The Add button is also a slot (slot="add-button"), which lets you inject arbitrary markup or an icon while preserving the built-in behaviour. 

<!-- Add button (light DOM slot) -->
  <button slot="add-button" data-action="add" class="add-button">+ Add</button>

  <!-- Custom buttons inside templates -->
  <template slot="display">
    <button data-action="toggle" title="Edit">✎</button>
    <button data-action="delete" title="Delete">🗑</button>
  </template>

CSS & theming

The component uses an open shadow DOM. It provides minimal built-in styles (a .hidden helper and modal overlay styling). All row layout and typography come from your templates and optional style[slot="styles"].

Shadow parts

  • ::part(root) – root container.
  • ::part(rows) – rows container.
  • ::part(add-button) – Add button container.
  • ::part(modal) – modal overlay (when modal-edit is used). Style the backdrop, background color, and blur effects here.
  • ::part(modal-surface) – modal dialog surface. Style the dialog box, borders, shadows, and animations here.
ck-editable-array {
  display: block;
  max-width: 720px;
  margin: 1rem auto;
}

ck-editable-array::part(root) {
  padding: 16px;
  background: #fff;
  border-radius: 8px;
}

/* Style modal overlay and surface */
ck-editable-array::part(modal) {
  background: rgba(0, 0, 0, 0.6);
  backdrop-filter: blur(4px);
}

ck-editable-array::part(modal-surface) {
  background: white;
  border-radius: 12px;
  padding: 24px;
  box-shadow: 0 20px 60px rgba(0, 0, 0, 0.3);
}

Quick example: Custom row styles

Use <style slot="styles"> inside your component to mirror styles into the shadow root and use parts for high-level host styling.

<ck-editable-array id="styled" name="tags">
  <style slot="styles">
    .row { padding: 10px; border-radius: 6px; background: #fff; }
    .row .label { font-weight: 600; }
  </style>
  <template slot="display">
    <div class="row"><span class="label" data-bind="name"></span></div>
  </template>
  <template slot="edit">...</template>
</ck-editable-array>

/* Host-level styling */
ck-editable-array::part(root) { box-shadow: 0 4px 14px rgba(0,0,0,0.08); }

Blazor (.razor)

<ck-editable-array id="peopleArray" @ref="arrayEl" name="person">...templates...</ck-editable-array>

@code {
  private ElementReference arrayEl;
  private List<Person> People = new() { new("Alice"), new("Bob") };

  protected override async Task OnAfterRenderAsync(bool firstRender) {
    if (firstRender) {
      await JS.InvokeVoidAsync("initEditableArray", arrayEl, People);
    }
  }
}

React

import '@colmkenna/ck-webcomponents/ck-editable-array';
import { useEffect, useRef } from 'react';

function People() {
  const ref = useRef(null);
  useEffect(() => {
    const el = ref.current;
    if (!el) return;
    el.data = [{ name: 'Alice' }, { name: 'Bob' }];
    const handler = (e) => console.log(e.detail.data);
    el.addEventListener('datachanged', handler);
    return () => el.removeEventListener('datachanged', handler);
  }, []);
  return (<ck-editable-array ref={ref} name="person">...templates...</ck-editable-array>);
}

Validation in modal edit

Assign a schema to enable validation. Use [data-field-error] for field messages and [data-error-summary] for row summaries. Inputs get aria-invalid when failing rules, and the Save button is disabled until the row is valid. 

<ck-editable-array id="ex" modal-edit>
  <template slot="edit">
    <div>
      <div data-error-summary role="alert" aria-live="polite"></div>
      <label>Name <input data-bind="name" /> <span data-field-error="name"></span></label>
      <label>Email <input data-bind="email" /> <span data-field-error="email"></span></label>
      <button data-action="cancel">Cancel</button>
      <button data-action="save">Save</button>
    </div>
  </template>
</ck-editable-array>

<script type="module">
  const ex = document.getElementById('ex');
  ex.schema = { required: ['name','email'], properties: { name: { minLength: 2 }, email: { minLength: 5 } } };
</script>

See the runnable modal demos above for complete working examples with validation.

Accessibility

  • Invalid inputs receive aria-invalid="true" and can be linked to data-field-error elements.
  • data-error-summary containers can be given role="alert" and aria-live for announcements.
  • Modal edit mode uses a dialog surface; ensure your templates provide clear labels and focus handling.

FAQ

Does it render anything without templates?

No. You must supply at least display and edit templates.

How do I submit to a server?

Set name on the component and use bound inputs; the generated names work with normal form posts.
